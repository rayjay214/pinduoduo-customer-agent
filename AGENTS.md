# AGENTS.md

## 工作方式

- 任何代码修改、配置修改、数据修复或上线相关操作，都必须先建立或更新根目录下的任务清单文档。
- 清单必须写清楚：要解决的问题、要做的任务、验证方式、执行记录。
- 执行修改时必须按清单推进，完成一项就在清单中打勾。
- 不允许只在对话里说明“改了什么”；必须把已做、未做、验证结果落到清单文档。
- 如果任务中途改变方向，先更新清单，再继续执行。
- 如果用户明确要求“不修改”，只允许审查、分析和更新用户要求的文档，不得改业务代码。

## 当前重点任务

- 公开仓库不提交真实商品链接、商品族代号、店铺数据、聊天记录、数据库、日志或内部执行清单。
- 涉及上线、发布或公开提交时，必须先做隐私扫描。

## RAG 预检索原则

- 大场景必须由订单状态、最新物流状态、客户是否确认收到共同决定。
- 没有订单信息时固定为售前。
- 有订单但未确认签收/收到时固定为售中。
- 订单状态或最新物流状态确认签收，或客户明确确认收到后，才进入售后。
- 客户文本关键词不能用于粗略猜测售前/售中/售后，只能用于已确定场景内的知识检索排序。
- 预检索应在确定的大场景知识库内做混合检索：向量召回、关键词/BM25 召回、别名/短语精确召回、meta 重排、命中质量闸门。
- 避免针对单个商品、单个会话、单个异常问法写死补丁；优先通过知识库 meta、配置化规则、prompt 约束或通用检索质量策略解决。

## 验证要求

- 代码改动后至少运行相关定向测试。
- 涉及 RAG、场景判定、工具调用或回复安全的改动，优先补充测试。
- 全量测试无法运行或失败时，必须在清单和最终回复中说明原因、失败点和剩余风险。
